# ข้อ 1 — The Buggy Payroll Logic (15 คะแนน)

โค้ดที่แก้แล้วอยู่ที่ [q1-payroll-fix.js](q1-payroll-fix.js)

## 3 จุดผิดพลาดร้ายแรง

### จุดที่ 1 — Floating Point (ทศนิยมเงินเพี้ยน)
**ปัญหา:** JavaScript ใช้ IEEE-754 double ทุกการคำนวณ เช่น `0.1 + 0.2 === 0.30000000000000004`
เมื่อนำมาคำนวณ `baseSalary * 0.05`, `baseSalary / 30 / 8 * 1.5` จะได้เศษทศนิยมยาว ๆ
สะสมข้ามพนักงาน 800 คนทุกเดือน → ยอดบัญชีไม่ตรง ตรวจสอบไม่ผ่าน

**วิธีแก้:** เก็บและคำนวณเงินเป็น **จำนวนเต็มหน่วยสตางค์** (1 บาท = 100 สตางค์)
และ `Math.round()` ทุกขั้นที่มีการหาร/คูณอัตรา เพื่อปัดให้ลงตัวที่ระดับสตางค์
(ทางเลือกอื่น: ใช้ `decimal.js` หรือชนิด `NUMERIC/DECIMAL` ในฐานข้อมูล — ห้ามใช้ `FLOAT`)

### จุดที่ 2 — SQL Injection
**ปัญหา:** โค้ดเดิมต่อสตริง `${net}` และ `${empId}` เข้า SQL โดยตรง
ถ้า `empId = "1 OR 1=1"` หรือ `"1; DROP TABLE salaries;--"` ฐานข้อมูลจะถูกทำลาย/รั่ว

**วิธีแก้:** ใช้ **parameterized query** (`$1`, `$2`) ให้ driver จัดการ escaping —
ค่าที่ส่งเข้าไปจะถูกมองเป็น "ข้อมูล" เสมอ ไม่ใช่ "คำสั่ง SQL"

### จุดที่ 3 — Race Condition (จ่ายเงินซ้ำ)
**ปัญหา:** ไม่มี transaction และไม่มีกลไกกันการรันซ้ำ ถ้า `processPayroll()` ของพนักงาน
คนเดียวกันถูกเรียกพร้อมกัน (กดปุ่มซ้ำ / API retry / รันงวดซ้ำ) เงินจะถูก `balance + net`
**สองครั้ง** → จ่ายเกิน

> หมายเหตุ: `UPDATE ... SET balance = balance + x` เป็น atomic ในระดับแถวก็จริง
> แต่ "ความซ้ำของการเรียก" คือปัญหาจริง — ต้องกันที่ระดับ "งวดถูก apply กี่ครั้ง"

**วิธีแก้:** 2 ชั้น
1. **Idempotency key** — ตาราง `payroll_runs` มี `UNIQUE(emp_id, payroll_run_id)`
   เรียกซ้ำงวดเดิม INSERT จะชนกัน → ข้ามการบวกเงิน
2. **Transaction + row lock** — `BEGIN` ... `SELECT ... FOR UPDATE` ... `COMMIT`
   ล็อกแถวกัน read-modify-write ซ้อนกันระหว่างทรานแซกชัน

## โบนัส (โดเมน HR ไทย)
ประกันสังคม 5% มี **เพดาน 750 บาท/เดือน** (ฐานเงินเดือนสูงสุด 15,000) โค้ดเดิมไม่ cap
จึงหัก SSO เกินจริงสำหรับเงินเดือนสูง → แก้ด้วย `Math.min(..., 750*100)`
