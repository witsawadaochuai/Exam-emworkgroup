Floating Point — เงินเพี้ยนจากเลขทศนิยม

เหตุผลที่ไม่ควรใช้ float คำนวณเงินโดยตรง

JavaScript เก็บเลขทศนิยมแบบ IEEE-754 double จึงไม่ได้แทนค่าทศนิยมบางค่าได้แม่นยำ 100%

0.1 + 0.2
// ค่าที่คาดหวัง: 0.3
// ค่าที่ได้จริง: 0.30000000000000004
กดปุ่มเพื่อดูผลลัพธ์

❌ ก่อนแก้: คำนวณเป็นบาทด้วย float

const sso = baseSalary * 0.05;
const otRate = (baseSalary / 30 / 8) * 1.5;
const net = baseSalary + (otHours * otRate) - sso;

การคูณและหารอาจสร้างเศษทศนิยมยาว เช่น 1234.999999 หรือ 1235.00000001 หากคำนวณพนักงาน 800 คนทุกเดือน ยอดรวมอาจไม่ตรงกับบัญชี

✅ หลังแก้: คำนวณเป็นสตางค์

const salarySatang = Math.round(baseSalary * 100);
const ssoSatang = Math.round(salarySatang * 0.05);

// เก็บและบันทึกเป็นจำนวนเต็ม
// 30,000 บาท = 3,000,000 สตางค์

กำหนดจุดปัดเศษด้วย Math.round() ให้ชัดเจน หรือใช้ decimal library และชนิด DECIMAL ในฐานข้อมูล

ทดลองกับเงินเดือน

ดูค่าดิบก่อนแก้ เทียบกับจำนวนสตางค์หลังแก้

กดคำนวณเพื่อเปรียบเทียบ
สรุป: เงินไม่ควรใช้ float คำนวณตรง ๆ ให้ใช้หน่วยเล็กสุด เช่น สตางค์ หรือใช้ decimal library / DECIMAL ในฐานข้อมูล

SQL Injection — เอาค่าจาก user มาต่อ SQL ตรง ๆ

ตัวอย่างการต่อข้อความ SQL เทียบกับ parameterized query

❌ ก่อนแก้

`UPDATE salaries
 SET balance = balance + ${net}
 WHERE emp_id = ${empId}`

ถ้า user ส่ง empId = "1 OR 1=1" SQL จะกลายเป็น:

UPDATE salaries
SET balance = balance + 1000
WHERE emp_id = 1 OR 1=1;

OR 1=1 เป็นจริงเสมอ จึง update พนักงานทุกคน

กรณีรุนแรง เช่น 1; DROP TABLE salaries;-- อาจลบตารางได้ หาก database หรือ driver อนุญาตให้รันหลายคำสั่ง

✅ หลังแก้

await db.query(
  `UPDATE salaries
   SET balance_satang = balance_satang + $1
   WHERE emp_id = $2`,
  [netSatang, empId]
);

Database driver รู้ว่า $1 และ $2 คือ “ข้อมูล” ไม่ใช่ “คำสั่ง SQL”

แม้ส่ง empId = "1 OR 1=1" ก็จะถูกค้นหาเป็น string ธรรมดา ไม่ได้เปลี่ยน logic ของ SQL

ทดลองกับข้อมูลพนักงาน 3 คน

ยอดที่จะเพิ่ม: 1,000 บาท · Payload: 1 OR 1=1

กดปุ่มเพื่อดูยอดเงินก่อนและหลังยิง

Race Condition — คำขอซ้ำทำให้จ่ายเงินซ้ำ

ทดลองยิง processPayroll พร้อมกัน 2 ครั้ง

กำหนดค่าที่ต้องการทดสอบ

ทุกคำขอใช้ Idempotency Key เดียวกัน เช่น PAYROLL-2026-06-EMP001

❌ ก่อนแก้

ทุกคำขอรัน balance = balance + net สำเร็จ จึงเพิ่มเงินซ้ำตามจำนวนที่ยิง

✅ หลังแก้

Transaction และ idempotency key ทำให้คำขอแรกสำเร็จ ส่วนคำขอที่เหลือถูกปฏิเสธว่าเป็นรายการซ้ำ

กำหนดตัวเลข แล้วเลือกทดลองก่อนแก้หรือหลังแก้
สรุป: transaction ช่วยควบคุมการทำงานพร้อมกัน ส่วน idempotency ป้องกันคำขอเดิมถูกประมวลผลซ้ำ

Night Shift SQL — หาคนกะดึกมาสาย

หาพนักงานกะ Night ของวันทำงาน 19 มี.ค. 2026 ที่มาสาย (สแกนหลัง 00:05)

กะ Night ทำงานช่วง 00:00–08:00 ของวันถัดไป แต่พนักงานมักสแกนนิ้วก่อนเที่ยงคืน (เช่น 23:55 ของวันก่อนหน้า) เพื่อเข้ากะ ทำให้ "วันที่บนนาฬิกา" กับ "วันทำงานของกะ" ไม่ตรงกัน

โจทย์: หาคนกะ Night ของ วันทำงาน 2026-03-19 ที่มาสาย = สแกนครั้งแรกหลัง 00:05

❌ ก่อนแก้

WHERE DATE(clock_in) = '2026-03-19'
  AND TIME(clock_in) > '00:05'

กรองด้วย "วันที่ = 19" ตรง ๆ จึงตัดคนที่สแกน 23:55 ของวันที่ 18 ทิ้ง และเพราะไม่ได้ดู "สแกนครั้งแรก" จึงเอาแถวเวลาอื่น (เช่น 04:00) มาตัดสินว่าสาย

✅ หลังแก้

WHERE clock_in >= '2026-03-18 23:00'
  AND clock_in <  '2026-03-19 08:00'
GROUP BY emp_id
HAVING MIN(clock_in) > '2026-03-19 00:05'

ใช้ ช่วงเวลาของวันทำงาน (เย็นวันก่อน → จบกะ) แล้วดู MIN(clock_in) = การสแกนครั้งแรก จึงรองรับการสแกนก่อนเที่ยงคืนได้ถูกต้อง

ทดลองกับข้อมูลสแกนนิ้ว

กะ Night วันทำงาน 2026-03-19 · เกณฑ์สาย = สแกนครั้งแรกหลัง 00:05

Emp IDชื่อเวลาสแกนหมายเหตุ
กดปุ่มเพื่อดูว่าใครถูกระบุว่ามาสาย
สรุป: งานกะข้ามเที่ยงคืนต้องคิดด้วย "วันทำงาน (business date)" ไม่ใช่วันที่บนนาฬิกา และใช้การสแกนครั้งแรก (MIN) ในการตัดสินมาสาย
